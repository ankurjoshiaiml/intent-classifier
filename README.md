# PAX Identity EKG — Intent Classifier V2

A production-grade intent classification pipeline for the **PAX Identity Enterprise Knowledge Graph (EKG)** system. It analyses natural-language messages from SAP users, classifies their intent (e.g. access request, SoD check, role design), extracts structured SAP entities, and enforces persona-based access gates — all driven by a master Excel definition sheet.

## Overview

When a user sends a message like **"Please give me access to ZC:P2P:PO_CREATOR________:1000"**, the system:

1. Retrieves the most similar example queries from a FAISS vector store
2. Builds a structured prompt injected with the intent taxonomy and retrieved examples
3. Calls **Claude Opus** (via AWS Bedrock) to classify the message
4. Post-processes the LLM response with deterministic regex-based entity extraction
5. Enforces a hard **persona gate** (e.g. END_USERs cannot run what-if scenarios)
6. Returns a rich `ClassificationResult` with: intent ID, sub-intent, confidence, extracted entities, bot response template, and reasoning

## The Intent Definition Sheet (V3.0)

The file `data/intent_definitions_v3.xlsx` (sheet name: **V3.0**) is the single source of truth for the entire system. Every intent, sub-intent, allowed persona, sample query, bot response template, and constraint is read from this file.

| Column | Name | Description |
|--------|------|-------------|
| A | Sl No. | Row number (1–6) |
| B | Common Intent Name | The intent label (e.g. "Validate user request") — becomes the `intent_id` |
| C | Action Intended | Numbered list of sub-intents (e.g. "1) Access Request\n2) Leavers...") |
| D | Persona % Coverage | Which personas can use this intent (e.g. "End User ~20%") |
| E | Sample Query | Numbered example user messages for vector embedding |
| F | Bot Responses | Numbered expected chatbot response templates |
| G | Constraints | Business rules for this intent |
| H | Comments | Notes for developers |

**Each row = one top-level intent group with multiple sub-intents, sample queries, and bot responses.**

## Module Breakdown

### `intent_registry.py`

**Purpose:** Parse the Excel sheet into structured Python objects.

**Key classes:**

| Class | Description |
|-------|-------------|
| `SubIntent` | One numbered entry from Column C (e.g. `index=1, label="Access Request"`) |
| `SampleQuery` | One numbered example from Column E with its sub-intent reference |
| `BotResponse` | One numbered response template from Column F |
| `IntentRecord` | Full parsed row: name, ID, sub-intents, personas, queries, responses, constraints, derived entity types |
| `IntentRegistry` | Wrapper class; loads all records once at startup; provides lookup by `intent_id` or name |

**Key functions:**

- `parse_intent_registry(excel_path)` — reads the V3.0 sheet and returns a list of `IntentRecord` objects
- `_parse_sub_intents(raw)` — parses `"1) Label\n2) Label"` patterns from column C
- `_parse_personas(raw)` — maps free-text persona descriptions to canonical codes (`END_USER`, `PROCESS_OWNER`, `DATA_OWNER`, `APP_OWNER`, `AUDITOR`)
- `_parse_numbered_items(raw)` — handles `"1)"`, `"1.1)"`, `"2)"` indexed multi-line cells
- `_derive_entity_types(intent_name, sub_intents)` — automatically infers which entity types (`SAP_ROLE`, `TCODE`, `USERNAME`, etc.) are relevant to each intent based on keywords

**The `intent_id`** is derived from the intent name by uppercasing and replacing spaces with `_` (e.g. `"Validate user request"` → `VALIDATE_USER_REQUEST`).

---

### `vector_store_v2.py`

**Purpose:** Embed every sample query from the registry into a FAISS vector index for semantic similarity search.

**How it works:**

1. **Document creation** — Each `SampleQuery` in the registry becomes an `EmbeddedDocument` carrying its full metadata: intent ID, sub-intent label, allowed personas, entity types, and the matching bot response template.
2. **Synthetic augmentation** — ~25 hand-crafted paraphrases (natural Teams-style phrasings) are added to improve recall for queries not literally in the sheet.
3. **Embedding** — Uses `sentence-transformers/all-MiniLM-L6-v2` (local, free, fast) to produce 384-dimensional normalised vectors.
4. **Indexing** — Stored in a FAISS `IndexFlatIP` (inner-product similarity = cosine similarity on normalised vectors).
5. **Persistence** — Saved to `data/vector_store_v2/` as `index.faiss` + `documents.pkl`.

**Search at runtime:**

```
store.search(query, k=5, persona_filter="END_USER")
```

- Encodes the query
- Fetches top `k*4` candidates
- Filters by persona (keeps examples the current user's persona can see)
- Returns top K `(EmbeddedDocument, score)` pairs

### `entity_extractor.py`

**Purpose:** Extract structured SAP entities from user messages using regex first, LLM second.

**Entity types and how they are detected:**

| Entity Type | Detection Method | Example |
|-------------|-----------------|---------|
| `SAP_ROLE` | Regex: `ZC:P2P:PO_CREATOR________:1000` | Part of ZC: namespace pattern |
| `TCODE` | Regex: `ME21N`, `MIGO`, `FB60`, `PFCG`... | Standard SAP transaction codes |
| `AUTH_OBJECT` | Regex: `M_BEST_EKG`, `Z_XXX_YYY` patterns | M_ and Z_ prefixed objects |
| `SOD_RULE_ID` | Regex: `SoD001`, `Rule 0001`, `SoD_0001` | Rule reference patterns |
| `USERNAME` | Regex on context: `"user pjunker"`, `"for pklasson"`, email/UPN | Context-aware to avoid false matches |
| `COUNTRY` | Keyword lookup: `"uk"` → `"United Kingdom"`, `"Germany"` → `"Germany"` etc. | ~15 countries mapped |
| `DEPARTMENT` | Keyword lookup: `"procurement"` → `"Procurement & Supply Chain"` etc. | ~10 dept groups mapped |
| `ACTIVITY` | Regex: `ACTVT=01`, `ACTVT:02` | Authorization field values |

**LLM fallback (Claude Opus via Bedrock):**

When regex finds no `SAP_ROLE` or `TCODE` for a task-based intent (e.g. `"I need to create purchase orders"`), the extractor sends a targeted prompt to Claude with SAP domain knowledge mapping tasks to TCodes:
- `"create purchase orders"` → `TCODE: ME21N, ME22N`
- `"approve purchase orders"` → `TCODE: ME28, ME29N`
- `"goods receipt"` → `TCODE: MIGO`
- `"invoice posting"` → `TCODE: MIRO`

**Merge strategy:** Regex results win over LLM results for structured patterns (roles, TCodes, auth objects). LLM fills gaps for semantic/free-text entities (departments, task descriptions).

### `prompt_builder_v2.py`

**Purpose:** Construct the system and user prompts sent to Claude Opus.

**System prompt contains:**
- A description of the PAX EKG system
- The full **intent taxonomy** for the current persona (intent IDs, names, sub-intents from Col C, allowed personas, entity types)
- The JSON output schema (intent_id, sub_intent, confidence, extracted_entities, persona_gate_passed, requires_clarification, clarification_question, reasoning, bot_response_template)
- Hard rules: persona gate, confidence threshold, strict JSON-only output

**User prompt contains:**
- Authenticated user context (username, persona, level, department, country)
- Up to 5 retrieved few-shot examples from the vector store (query text + intent + sub-intent + expected bot response)
- The actual message to classify

**`get_allowed_records_for_persona(registry, persona)`** filters the intent taxonomy to only show intents the current persona is allowed to see, preventing prompt injection-style leakage of privileged intent options.

### `classifier_v2.py`

**Purpose:** Orchestrate the full pipeline and return a `ClassificationResult`.

**`ClassificationResult` fields:**

| Field | Type | Description |
|-------|------|-------------|
| `intent_id` | str | Machine-safe intent identifier (e.g. `DETECT_ROLES_REQUESTED`) |
| `intent_name` | str | Human-readable intent name |
| `sub_intent` | str | The specific Action Intended sub-category matched |
| `confidence` | float | 0.0–1.0 confidence score from the LLM |
| `extracted_entities` | dict | All found entities (SAP roles, TCodes, usernames, etc.) |
| `persona_gate_passed` | bool | Whether the user's persona is allowed for this intent |
| `requires_clarification` | bool | True if confidence < 0.70 or LLM flagged ambiguity |
| `clarification_question` | str | Follow-up question to ask the user |
| `reasoning` | str | 1–2 sentence explanation from the LLM |
| `bot_response_template` | str | Closest Col F bot response for the matched sub-intent |
| `processing_time_ms` | float | End-to-end latency in milliseconds |
| `retrieval_info` | list | Top-3 retrieved examples with scores (for debugging) |

**8-step pipeline (per `classify()` call):**

```
1. Filter registry → only intents allowed for this persona
2. Vector search → top-5 similar examples from FAISS
3. Build system_prompt (taxonomy + schema) + user_prompt (context + examples)
4. Call AWS Bedrock (Claude Opus) → raw JSON string
5. Parse & validate JSON response
6. Regex entity extraction → merge with LLM-extracted entities
7. Hard persona gate → override to OUT_OF_SCOPE if blocked
8. Confidence gate → set requires_clarification if score < 0.70
```

## Classification Pipeline (Step by Step)

Here is a concrete walkthrough for:
> **"Please give me access to ZC:P2P:PO_CREATOR________:1000"** — Persona: END_USER

```
Step 1 — Allowed intents for END_USER:
   VALIDATE_USER_REQUEST, DETECT_ROLES_REQUESTED, CHECK_FOR_SOD,
   ELABORATE_ROLE_FUNCTIONS, KNOWLEDGE_GRAPH_BASED_SCENARIOS

Step 2 — Vector search:
   Encodes query → searches FAISS index → returns top 5 examples
   e.g. "Can I get the PO creator role?" (score 0.91, DETECT_ROLES_REQUESTED)
        "I need to be able to create purchase orders" (score 0.87, DETECT_ROLES_REQUESTED)

Step 3 — Build prompts:
   system = intent taxonomy filtered for END_USER + JSON schema
   user = username/persona context + 5 examples + the message

Step 4 — Bedrock call:
   Model: Claude Opus (eu.anthropic.claude-opus-4-5...)
   Max tokens: 1024

Step 5 — LLM returns JSON:
   {
     "intent_id": "DETECT_ROLES_REQUESTED",
     "sub_intent": "Access Request (with preventive SoD)",
     "confidence": 0.95,
     "extracted_entities": {"SAP_ROLE": ["ZC:P2P:PO_CREATOR________:1000"]},
     "persona_gate_passed": true,
     "requires_clarification": false,
     "reasoning": "User explicitly requests a named SAP role."
   }

Step 6 — Regex extraction:
   SAP_ROLE_PATTERN matches: ZC:P2P:PO_CREATOR________:1000 ✓
   Merged with LLM entities (regex wins for SAP_ROLE)

Step 7 — Persona gate:
   DETECT_ROLES_REQUESTED allows END_USER ✓ — no override

Step 8 — Confidence gate:
   0.95 >= 0.70 ✓ — no clarification needed

Final result: intent_id=DETECT_ROLES_REQUESTED, SAP_ROLE=[...], confidence=0.95
```

## Intent Taxonomy

The system recognises **6 core intents** plus an `OUT_OF_SCOPE` catch-all:

| # | Intent ID | Description | Key Sub-intents |
|---|-----------|-------------|-----------------|
| 1 | `VALIDATE_USER_REQUEST` | New joiner/leaver/access review for a user | Access Request, Leavers Revoke, Self Review |
| 2 | `DETECT_ROLES_REQUESTED` | User wants a specific role or asks what roles peers have | Access Request, Access Request (with preventive SoD) |
| 3 | `CHECK_FOR_SOD` | SoD conflict detected, remediation required | Preventive SoD check, Remediation (Auditor), Mitigation |
| 4 | `ELABORATE_ROLE_FUNCTIONS` | Explain what a role does; design new roles | Role Explanation, Role Design (New/Existing) |
| 5 | `KNOWLEDGE_GRAPH_BASED_SCENARIOS` | Graph queries: who has what access, access reviews, risk maps | Access Review (Line Manager), Risk Assessment (Auditor) |
| 6 | `RUN_A_WHATIF_SCENARIO` | Simulate role/permission changes to see SoD impact | What-If (existing role), Role Design Simulation |
| — | `OUT_OF_SCOPE` | Query unrelated to SAP access/SoD/identity governance | — |

| Entity | Description | Examples |
|--------|-------------|---------|
| `SAP_ROLE` | Technical SAP role names | `ZC:P2P:PO_CREATOR________:1000` |
| `TCODE` | SAP transaction codes | `ME21N`, `MIGO`, `MIRO`, `FB60` |
| `AUTH_OBJECT` | SAP authorization objects | `M_BEST_EKG`, `M_EINK_FRG` |
| `SOD_RULE_ID` | SoD rule references | `SoD001`, `Rule 0032` |
| `USERNAME` | SAP user ID or UPN | `pjunker`, `user@company.com` |
| `COUNTRY` | Country the request relates to | `United Kingdom`, `Germany` |
| `DEPARTMENT` | Business function/department | `Procurement & Supply Chain`, `Finance` |
| `ACTIVITY` | SAP ACTVT authorization values | `01` (create), `02` (edit) |

## Persona System

The system enforces **five personas**. Each intent in the V3.0 sheet specifies which personas are allowed:

| Persona | Code | Typical Access |
|---------|------|---------------|
| End User / Line Manager | `END_USER` | Own access, peer roles, SoD notifications |
| Process / Role Owner | `PROCESS_OWNER` | Role design, access reviews, SoD remediation |
| Data Owner | `DATA_OWNER` | Data-level access reviews |
| Application Owner | `APP_OWNER` | Role design, technical configuration |
| Auditor | `AUDITOR` | Full read access including risk reports, what-if |

**Persona gate:** If a user with persona `END_USER` asks a question mapped to `RUN_A_WHATIF_SCENARIO` (which requires `PROCESS_OWNER` or `APP_OWNER`), the classifier hard-overrides the result to `OUT_OF_SCOPE` and returns an explanatory clarification message.

## Setup & Usage

### Prerequisites

- Python 3.10+
- AWS account with Bedrock access enabled for Claude Opus in `eu-west-2`
- The `data/intent_definitions_v3.xlsx` file in position

### 1. Create virtual environment and install dependencies

```bash
python3 -m venv env
source env/bin/activate          # macOS/Linux
pip install -r requirements.txt
```

### 2. Configure environment variables

Create a `.env` file in the project root (see [Configuration](#configuration-env)):

```env
AWS_ACCESS_KEY_ID=your_key
AWS_SECRET_ACCESS_KEY=your_secret
AWS_DEFAULT_REGION=eu-west-2
BEDROCK_MODEL_ID=arn:aws:bedrock:eu-west-2:613564179512:inference-profile/eu.anthropic.claude-opus-4-5-20251101-v1:0
```

### 3. Build the vector store (one-time setup)

```bash
python main_v2.py setup
```

This will:
- Parse the V3.0 Excel sheet
- Embed all sample queries with `all-MiniLM-L6-v2`
- Save FAISS index to `data/vector_store_v2/`

### 4. Run the interactive REPL

```bash
python main_v2.py run
```

Type any natural language query. Use `persona` to switch between END_USER / PROCESS_OWNER / AUDITOR etc. Type `quit` to exit.

### 5. Run the test suite

```bash
python main_v2.py test
```

Runs 13 predefined test cases covering all 6 intents plus persona gate validation. Outputs PASS/FAIL with details.

### 6. Show parsed intent registry

```bash
python main_v2.py show
```

Prints all parsed intents with their sub-intents, personas, entity types, and query counts.
